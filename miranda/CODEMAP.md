# Miranda Code Map

This file is a quick guide for navigating the Miranda implementation in this
directory.

## High-Level Pipeline

1. Parse source (`rules.y`, `lex.c`) into internal expression trees.
2. Translate language constructs into combinatory logic (`trans.c`).
3. Evaluate with the reducer machine (`reduce.c`).
4. Allocate/evolve runtime objects in the custom heap (`data.c`, `big.c`).

## Core Source Files

- `steer.c`: process startup, CLI flags, session command loop, script loading,
  startup policy (`.mirarc`, prelude/stdenv wiring), and top-level control flow.
- `lex.c` / `rules.y`: lexer and parser grammar; parser drives declarations and
  expression evaluation requests.
- `trans.c`: code generation from parsed Miranda forms into internal
  combinator-based runtime form.
- `reduce.c`: runtime evaluator (reduction machine), built-ins, forcing rules,
  output message handling.
- `data.c`: heap layout, allocation primitives, garbage collection, object
  serialization helpers.
- `big.c`: bignum/integer support used across parser/runtime.
- `types.c`: type checker and type environment handling.
- `signals.c`: signal/interrupt support.
- `utf8.c`: UTF-8 support functions used by lexer/runtime I/O paths.

## Data Model (Conceptual)

- Runtime objects are tagged nodes (`tag[]`) with head/tail payload fields
  (`hd`, `tl`) allocated in the custom heap.
- Lists, applications, constructors, primitives, atoms, numbers, and parser
  artifacts share this representation via tag-based dispatch.
- The reducer assumes expression trees are in this runtime representation and
  mutates/evaluates them in place in key hot paths.

## Memory and GC

- Heap allocation is managed in `data.c` via `make(tag, hd, tl)`.
- GC is conservative with explicit roots plus stack scanning.
- Several modules expose root pointers/globals used during GC; changing those
  paths requires validating that all live references remain reachable at GC.
- Optimization flags can change stack/register behavior, which is why GC
  stability notes in `README.md` and `Makefile` matter.

## Build Artifacts and Generated Files

- `y.tab.c`, `y.tab.h`: generated from `rules.y` by `byacc`.
- `cmbnms.c`, `combs.h`: generated by `gencdecs`.
- `mira`: main executable.
- `miralib/`: runtime library, examples, and manual assets loaded by `mira`.

## Entry Points Worth Reading First

1. `steer.c:main`
2. `data.c:make`, `data.c:gc`
3. `trans.c:codegen`, `trans.c:abstract` / `abstr`
4. `reduce.c:output`, `reduce.c:reduce`, `reduce.c:force`

## Practical Change Tips

- Prefer adding diagnostics near call boundaries (`steer` -> `trans` -> `reduce`)
  before touching GC-sensitive data paths.
- For parser or grammar changes, regenerate parser files with `byacc` and test
  both compile-time and runtime behavior.
- For runtime semantic changes, test both direct expression evaluation and
  script execution because code paths differ in command vs script mode.
